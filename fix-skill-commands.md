# Allowlist mode vs agent-generated commands

## Problem

OpenClaw's `allowlist` mode rejects commands with chaining (`&&`, `;`, `||`) and redirections (`2>/dev/null`, `>`, `>>`) unless **every segment** satisfies the allowlist.

From [OpenClaw docs](https://docs.openclaw.ai/tools/exec):
> "Chaining (`;`, `&&`, `||`) and redirections are rejected in allowlist mode unless every top-level segment satisfies the allowlist."

**The agent itself generates these patterns** — `cd && ...`, `2>/dev/null`, `|| echo` — so this cannot be fixed in skill code. It's a fundamental incompatibility between `allowlist` mode and how the agent constructs shell commands.

## Examples of agent-generated commands that get blocked

```bash
# cd && chaining — cd is a shell built-in, cannot be allowlisted
cd /home/bzsega/.openclaw/workspace/skills/sergei-mikhailov-stt && python3 scripts/stt_processor.py --file "$AUDIO_FILE"
cd /home/bzsega/.openclaw/workspace/skills/ai-models-monitor && node index.js --period 24h

# Redirection
find ~/.openclaw/workspace -name "*ai*model*" -type f 2>/dev/null

# Redirection + chaining
ls -la /home/bzsega/.openclaw/workspace/*ai*model* 2>/dev/null || echo "No files found"
```

All binaries (`python3`, `node`, `find`, `ls`) are in the allowlist. The prompts appear because of `&&`, `||`, `2>/dev/null`.

## Workaround

Switch from `allowlist` to `full` security mode:

```bash
openclaw approvals set defaults.security full
systemctl --user restart openclaw-gateway.service
```

This allows all commands without prompts. The allowlist entries are preserved and can be re-enabled when OpenClaw fixes chaining support in allowlist mode.

## Why not just fix the commands?

The commands are generated by the agent at runtime, not hardcoded in skill source files. The agent naturally uses:
- `cd dir && command` — to set working directory
- `2>/dev/null` — to suppress error output
- `cmd1 || cmd2` — for fallback handling

These are standard shell patterns that any LLM agent will produce. The `allowlist` mode doesn't account for this.
